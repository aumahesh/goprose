# goprose

Go implementation of the ProSe compiler. Takes `.prose` source files written in guarded-command notation and generates a complete Go module (proto definitions, interface, implementation, main, Makefile).

## Build and test

```bash
make build          # builds bin/prose
make test           # runs go test ./...
go test ./...       # same
```

## Running the compiler

```bash
bin/prose -p proseFiles/max.prose -o _examples/
# or during development:
go run cmd/prose.go -p proseFiles/max.prose -o _examples/
```

The `-p` flag is the source prose file; `-o` is the output directory. The compiler writes `<output>/<ProgramName>/` containing a ready-to-build Go module.

## Repository layout

```
cmd/               – compiler entry point (prose.go) and integration tests
internal/
  parser/          – participle-based parser; grammar types in prose.go
  intermediate/    – AST → intermediate representation + code generation
  templates/       – template renderer (render.go)
  util/            – camelCase helper
templates/         – Go text/template files rendered per program
proseFiles/        – example .prose source files
_examples/         – generated Go modules (committed, regenerated by tests)
```

## Compilation pipeline

```
.prose file
  → parser.Parse()              (internal/parser)
  → intermediate.GenerateIntermediateProgram()   (internal/intermediate)
  → templates.Render()          (internal/templates, templates/*.tmpl)
  → generated Go module
```

### Parser (`internal/parser/prose.go`)

Uses [participle v2](https://github.com/alecthomas/participle) with `UseLookahead(4)`. Grammar types are plain Go structs with participle struct tags. The canonical grammar is documented in the block comment at the top of `prose.go`.

Key types:
- `Program` – top-level; holds variable/const declarations and `Statements`
- `Statement` – union: `Alternative` (`if…fi`) | `Repetitive` (`do…od`) | `Guarded` (original ProSe form)
- `GuardedProSeStatement` – `[<priority>] guard -> action+`
- `AlternativeConstruct` / `RepetitiveConstruct` – Dijkstra GCL constructs
- `GuardedCommand` – `guard -> GclAction+` (used inside GCL constructs)
- `GclAction` – union: assignment | `AlternativeConstruct` | `RepetitiveConstruct`

### Intermediate representation (`internal/intermediate/`)

Translates the parsed AST into `Program` (a flat struct with maps of variables/constants and a slice of `GuardedStatement`s). `GuardedStatement` holds pre-generated Go code lines in `GuardCode` and `ActionCode`.

Key entry points:
- `GenerateIntermediateProgram(*parser.Program) (*Program, error)`
- `NewStatement(index, *parser.Statement, …) (*GuardedStatement, error)` – dispatches on the Statement union
- `NewGclAction(*parser.GclAction, …) (*Action, error)` – generates code for assignments and nested GCL constructs

Expression code generation lives in `expression.go`. Expressions are classified as `ExpressionTypeLocal` or `ExpressionTypeRemote` (references neighbor state); remote guards generate a `for _, neighbor = range this.neighborState { … }` loop.

Temp variable names (`temp0`, `temp1`, …) are allocated from `TempsManager` which guarantees uniqueness across a whole compilation unit.

### Templates (`templates/`)

Six templates are rendered per program:
- `impl_template_go.tmpl` – the implementation struct; priority helpers; `evaluateGuardN`/`executeActionN` function pairs; `updateLocalState`; networking boilerplate
- `interface_template_go.tmpl` – the interface type
- `main_template_go.tmpl` – `main()` entry point
- `state_proto.tmpl` – protobuf state message
- `go_mod_template.tmpl` – `go.mod`
- `makefile.tmpl` – per-module Makefile

Template functions are registered in `render.go`: `getType`, `isString`, `getConstantType`, `isConstantString`, `increment`, `protoName`.

## Priority scheduling

Each statement has a `configuredPriority` (default 1, set by `<N>` prefix). On each call to `updateLocalState()`:

1. `evaluateGuardN()` is called for every statement. It **peeks** at `currentPriority - 1` without mutating state.
2. From the guards that returned true, one is picked at random.
3. `executeActionN()` **decrements** the running priority. If it reached 0 the action runs and the priority resets.

This separation (peek in guard, decrement in action) means guard evaluation has no side effects.

## GCL constructs

`if…fi` and `do…od` are valid top-level statements. Both get a trivial outer guard (`takeAction = true`) so they participate in priority scheduling identically to regular guarded statements.

- **`if…fi`**: evaluates all internal guards, collects true ones, picks one with `rand.Intn`. No-op if none are true.
- **`do…od`**: same selection in a `for` loop; breaks when no guard is true.

GCL construct guards must reference only local variables (remote/neighbor variables are not supported inside `if…fi` or `do…od`).

## Adding a new `.prose` example

1. Write `proseFiles/<name>.prose`.
2. Add `"<name>.prose"` to `proseFiles` slices in both:
   - `internal/parser/parse_test.go` (`TestProSeParser_Parse`)
   - `cmd/prose_test.go` (`TestCompileAndGenerateCode`)
3. Run `go test ./...`; the codegen test writes the generated module to `_examples/`.
4. Commit both the prose file and the generated `_examples/<ProgramName>/`.

## Changing the template

Edit `templates/impl_template_go.tmpl`. Re-run `go test ./cmd/...` to regenerate all `_examples/` and catch regressions. The examples are committed so diffs are visible in review.

## Changing the parser grammar

Edit the struct tags in `internal/parser/prose.go`. If you add a new keyword (`if`, `do`, `fi`, `od`, …) it does not need to be added to the lexer — participle matches `Ident` tokens against literal strings in grammar tags.

After parser changes, add or update a `.prose` test file and verify with `go test ./internal/parser/...`.
