# goprose

Go implementation of the ProSe compiler. Takes `.prose` source files written in guarded-command notation and generates a complete Go module (proto definitions, interface, implementation, main, Makefile).

## Tools

| Binary | Purpose |
|--------|---------|
| `bin/prose` | Compiler: `.prose` ‚Üí Go module |
| `bin/prose-sim` | Visualizer: runs the algorithm in-process, serves a web UI |

## Build and test

```bash
make build          # builds bin/prose
make build-sim      # builds bin/prose-sim
make test           # runs go test ./...
go test ./...       # same
```

## Running the compiler

```bash
bin/prose -p proseFiles/max.prose -o _examples/
# or during development:
go run cmd/prose.go -p proseFiles/max.prose -o _examples/
```

The `-p` flag is the source prose file; `-o` is the output directory. The compiler writes `<output>/<ProgramName>/` containing a ready-to-build Go module.

## Running the simulator

```bash
# auto-generated topology
bin/prose-sim -p proseFiles/gcd.prose --nodes 4 --topology ring

# YAML topology file
bin/prose-sim -p proseFiles/gcd.prose --topo-file topology.yaml

# per-node state override (repeatable)
bin/prose-sim -p proseFiles/pursuer_evader_with_priority.prose \
  --nodes 10 --topology random \
  --node-state n5.isEvaderHere=true
```

Open `http://localhost:8080`. All flags:

| Flag | Default | Description |
|------|---------|-------------|
| `-p` | *(required)* | Path to `.prose` file |
| `--nodes` | `3` | Number of nodes (with `--topology`) |
| `--topology` | `ring` | `ring`, `fully-connected`, or `random` |
| `--topo-file` | ‚Äî | YAML topology file (overrides `--nodes`/`--topology`) |
| `--node-state` | ‚Äî | `<nodeID>.<varName>=<value>`, repeatable |
| `--addr` | `:8080` | HTTP listen address |
| `--templates` | `templates/` | Path to templates dir (for "Generate Go" button) |

`random` topology is always connected ‚Äî it builds a random spanning tree first, then adds extra edges.

## Compilation pipeline

```
.prose file
  ‚Üí parser.Parse()              (internal/parser)
  ‚Üí intermediate.GenerateIntermediateProgram()   (internal/intermediate)
  ‚Üí templates.Render()          (internal/templates, templates/*.tmpl)
  ‚Üí generated Go module
```

### Parser (`internal/parser/prose.go`)

Uses [participle v2](https://github.com/alecthomas/participle) with `UseLookahead(4)`. Grammar types are plain Go structs with participle struct tags. The canonical grammar is documented in the block comment at the top of `prose.go`.

Key types:
- `Program` ‚Äì top-level; holds variable/const declarations and `Statements`
- `Statement` ‚Äì union: `Alternative` (`if‚Ä¶fi`) | `Repetitive` (`do‚Ä¶od`) | `Guarded` (original ProSe form)
- `GuardedProSeStatement` ‚Äì `[<priority>] guard -> action+`
- `AlternativeConstruct` / `RepetitiveConstruct` ‚Äì Dijkstra GCL constructs
- `GuardedCommand` ‚Äì `guard -> GclAction+` (used inside GCL constructs)
- `GclAction` ‚Äì union: assignment | `AlternativeConstruct` | `RepetitiveConstruct`

### Intermediate representation (`internal/intermediate/`)

Translates the parsed AST into `Program` (a flat struct with maps of variables/constants and a slice of `GuardedStatement`s). `GuardedStatement` holds pre-generated Go code lines in `GuardCode` and `ActionCode`.

Key entry points:
- `GenerateIntermediateProgram(*parser.Program) (*Program, error)`
- `NewStatement(index, *parser.Statement, ‚Ä¶) (*GuardedStatement, error)` ‚Äì dispatches on the Statement union
- `NewGclAction(*parser.GclAction, ‚Ä¶) (*Action, error)` ‚Äì generates code for assignments and nested GCL constructs

Expression code generation lives in `expression.go`. Expressions are classified as `ExpressionTypeLocal` or `ExpressionTypeRemote` (references neighbor state); remote guards generate a `for _, neighbor = range this.neighborState { ‚Ä¶ }` loop.

Temp variable names (`temp0`, `temp1`, ‚Ä¶) are allocated from `TempsManager` which guarantees uniqueness across a whole compilation unit.

### Templates (`templates/`)

Six templates are rendered per program:
- `impl_template_go.tmpl` ‚Äì the implementation struct; priority helpers; `evaluateGuardN`/`executeActionN` function pairs; `updateLocalState`; networking boilerplate
- `interface_template_go.tmpl` ‚Äì the interface type
- `main_template_go.tmpl` ‚Äì `main()` entry point
- `state_proto.tmpl` ‚Äì protobuf state message
- `go_mod_template.tmpl` ‚Äì `go.mod`
- `makefile.tmpl` ‚Äì per-module Makefile

Template functions are registered in `render.go`: `getType`, `isString`, `getConstantType`, `isConstantString`, `increment`, `protoName`.

## Priority scheduling

Each statement has a `configuredPriority` (default 1, set by `<N>` prefix). On each call to `updateLocalState()`:

1. `evaluateGuardN()` is called for every statement. It **peeks** at `currentPriority - 1` without mutating state.
2. From the guards that returned true, one is picked at random.
3. `executeActionN()` **decrements** the running priority. If it reached 0 the action runs and the priority resets.

This separation (peek in guard, decrement in action) means guard evaluation has no side effects.

## GCL constructs

`if‚Ä¶fi` and `do‚Ä¶od` are valid top-level statements. Both get a trivial outer guard (`takeAction = true`) so they participate in priority scheduling identically to regular guarded statements.

- **`if‚Ä¶fi`**: evaluates all internal guards, collects true ones, picks one with `rand.Intn`. No-op if none are true.
- **`do‚Ä¶od`**: same selection in a `for` loop; breaks when no guard is true.

GCL construct guards must reference only local variables (remote/neighbor variables are not supported inside `if‚Ä¶fi` or `do‚Ä¶od`).

## Simulator internals (`internal/sim/`, `internal/viz/`)

### Packages

- **`internal/sim/eval.go`** ‚Äî runtime expression evaluator. Walks the `parser.Expr` AST and returns `int64`/`bool`/`string` values. Mirrors the code-generation walk in `intermediate/expression.go`.
- **`internal/sim/engine.go`** ‚Äî `Node` and `Engine` types. `Engine.Step()` picks a uniform-random node, evaluates all eligible guards, picks one true guard at random, and executes its action.
- **`internal/sim/topology.go`** ‚Äî `LoadTopology(yamlFile)` and `GenerateTopology(n, style, rng)`. `random` style guarantees connectivity via a spanning tree.
- **`internal/viz/server.go`** ‚Äî HTTP server. REST endpoints: `GET /api/state`, `POST /api/step`, `POST /api/reset`, `GET /api/compile` (returns zip of generated Go module).
- **`internal/viz/ui.go`** ‚Äî embedded single-page app (D3.js force graph, step/auto/reset controls, scrolling log).

### Parser/evaluator coupling

`eval.go` and `intermediate/expression.go` both walk the same `parser.Expr` AST. A grammar change (new expression type, new operator) must be reflected in **both** files. The integration tests in `internal/sim/sim_test.go` (GCD convergence, max propagation, if‚Ä¶fi, do‚Ä¶od, Reset) will catch drift.

`intermediate.Expression.GetExpr()` exposes the underlying `*parser.Expr` so the sim evaluator can reach init expressions (e.g. `= rand.Int63n(1000)`) without going through the code-generation path.

### Priority in the simulator

Each node maintains a per-statement countdown. On every call to `stepNode`:
- If `countdown > 0`: decrement and skip (not eligible this step).
- If `countdown == 0`: evaluate guard; after eligibility (whether or not selected), reset to `configuredPriority - 1`.

For priority 1 the countdown is always 0 ‚Äî fires every step. For priority N it fires once every N steps of that node.

## Adding a new `.prose` example

1. Write `proseFiles/<name>.prose`.
2. Add `"<name>.prose"` to `proseFiles` slices in both:
   - `internal/parser/parse_test.go` (`TestProSeParser_Parse`)
   - `cmd/prose_test.go` (`TestCompileAndGenerateCode`)
3. Run `go test ./...`; the codegen test writes the generated module to `_examples/`.
4. Commit both the prose file and the generated `_examples/<ProgramName>/`.

## Changing the template

Edit `templates/impl_template_go.tmpl`. Re-run `go test ./cmd/...` to regenerate all `_examples/` and catch regressions. The examples are committed so diffs are visible in review.

## Changing the parser grammar

Edit the struct tags in `internal/parser/prose.go`. If you add a new keyword (`if`, `do`, `fi`, `od`, ‚Ä¶) it does not need to be added to the lexer ‚Äî participle matches `Ident` tokens against literal strings in grammar tags.

After parser changes, add or update a `.prose` test file and verify with `go test ./internal/parser/...`.
